# Werka 状态页优化总计划

## 1. 目标

只做一件事：

把 Werka 里的这三个页面做快：

- 进行中
- 已确认
- 已退回

要求：

- 打开快
- 第一屏马上出现
- 能继续下滑加载
- 不靠一次性加载全部数据

## 2. 当前问题

现在慢的地方不是写入。

慢的是读取：

- 状态分组页
- 状态详情页

用户点进去后会等。

这不可以。

## 3. 不做什么

这次不做：

- Supplier 其它页面
- Admin 其它页面
- 写流程重构
- ERP 业务真相改动

只做 Werka 状态页速度。

## 4. 核心原则

原则只有四个：

1. 读走 DB
2. 写走 ERP API
3. 首屏只拿第一批
4. 搜索是辅助，不是强制

## 5. 正确模型

状态页必须这样工作：

### 分组页

- 点击后立刻打开
- 后端直接给分组结果
- 不先拉全量再分组

### 详情页

- 只拿第一页
- 滚动再拿下一页
- 不能一口气把所有记录全拿回来

## 6. 后端必须做的事

### 6.1 分组接口

为这三类状态做快速读取：

- pending
- confirmed
- returned

每条分组数据至少有：

- 对方 ID
- 对方名称
- 数量统计
- 发出总量
- 接收总量
- 退回总量
- UOM

### 6.2 详情接口

详情接口必须支持：

- limit
- offset

并且：

- 排序稳定
- 只返回当前页

### 6.3 数据来源

后端优先直接查 DB。

不要再走慢的 API 链式读取。

## 7. 前端必须做的事

### 7.1 分组页

- 首屏只等分组结果
- 有数据就先画出来

### 7.2 详情页

- 第一页先显示
- 滚动到底再继续拉
- 加载下一页时不能卡死整个页面

### 7.3 错误处理

- 第一页失败：显示 retry
- 后续页失败：只影响底部，不要清空已有列表

## 8. 状态语义不能错

速度再快也不能算错。

必须保证：

- Home 里的数字
- 状态分组里的数字
- 状态详情里的条目

三者逻辑一致。

## 9. 实施顺序

严格按这个顺序做：

1. 量当前接口耗时
2. 做 DB 分组查询
3. 做 DB 详情分页查询
4. 前端接分页
5. 前端加无限滚动
6. 校验数字是否一致
7. 真机测试

## 10. 验收标准

达到下面几点才算完成：

- 点“进行中”后明显快
- 点“已确认”后明显快
- 点“已退回”后明显快
- 第一屏不用等很久
- 下滑能继续加载
- 数字不乱
- 测试通过

## 11. 一句话总结

这次只做一件事：

把 Werka 的状态页改成：

`DB 直读 + 首屏快开 + 分页 + 无限滚动`
